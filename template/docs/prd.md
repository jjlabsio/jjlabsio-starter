## 2. 서비스 아키타입 판별

> **이 섹션을 가장 먼저 작성한다.** 아키타입에 따라 이후 섹션의 구조가 완전히 달라진다.
> 아래 표에서 가장 가까운 아키타입을 선택하거나, 복합형인 경우 조합을 명시한다.

### 2.1 아키타입 선택

| 아키타입                | 특징                                                                     | 해당 여부 |
| ----------------------- | ------------------------------------------------------------------------ | --------- |
| **대시보드 SaaS**       | 로그인 → 대시보드 → 데이터 조회/관리. 예: 분석 도구, CRM, 모니터링       |           |
| **단일 기능 도구**      | 입력 → 처리 → 결과. 로그인 없이도 쓸 수 있음. 예: 변환기, 생성기, 계산기 |           |
| **마켓플레이스/플랫폼** | 공급자 ↔ 소비자 양면. 예: 커뮤니티, 매칭, 거래 플랫폼                    |           |
| **브라우저 익스텐션**   | 기존 사이트 위에서 동작. 별도 웹앱은 설정/결제용. 예: 스팸 필터, 번역기  |           |
| **CLI / 개발자 도구**   | 터미널 기반. 웹은 문서/대시보드용. 예: SDK, CI/CD 도구                   |           |
| **모바일 앱 (PWA)**     | 모바일 퍼스트. 푸시/오프라인 중요. 예: 습관 트래커, 알림 서비스          |           |
| **API 서비스**          | 개발자가 연동. 웹은 문서/콘솔용. 예: AI API, 결제 API                    |           |
| **콘텐츠/미디어**       | 콘텐츠 발행 + 구독. 예: 뉴스레터, 큐레이션                               |           |
| **랜딩 Only**           | 제품 없이 랜딩으로 수요 검증만. 예: 웨이트리스트, 사전등록               |           |

**선택한 아키타입**: [예: 대시보드 SaaS + 브라우저 익스텐션 복합]

### 2.2 아키타입이 결정하는 것

> 선택한 아키타입에 따라 아래 항목이 달라진다. 해당하는 것만 체크한다.

| 결정 사항        | 이 서비스에서는                                                     |
| ---------------- | ------------------------------------------------------------------- |
| 핵심 인터페이스  | [예: 웹 대시보드 / 익스텐션 팝업 / CLI / 단일 페이지 / 모바일 등]   |
| 인증 필요 여부   | [예: 필수 / 핵심 기능은 비로그인, 저장만 로그인 / 불필요]           |
| 데이터 저장 방식 | [예: 서버 DB / 로컬스토리지 / 클라우드 동기화 / 세션 한정]          |

## 5. 서비스 구조 & 화면 설계

> **이 섹션은 2장에서 선택한 아키타입에 맞게 자유롭게 설계한다.**
> 아래는 설계 시 반드시 고려해야 할 체크리스트이며, 구조 자체는 강제하지 않는다.

### 5.1 사용자 여정 (핵심 플로우)

> 사용자가 서비스를 처음 만나는 순간부터 핵심 가치를 체감하기까지의 흐름을 단계별로 서술한다.

```
[첫 접점] → [가입/설치] → [초기 설정] → [핵심 액션] → [가치 체감] → [재방문/습관화]
```

| 단계         | 사용자 행동                          | 화면/인터페이스 | 이탈 리스크 | 대응 |
| ------------ | ------------------------------------ | --------------- | ----------- | ---- |
| 1. 첫 접점   | [예: 랜딩 방문 / 스토어에서 발견]    |                 |             |      |
| 2. 시작      | [예: 가입 / 설치 / 바로 사용]        |                 |             |      |
| 3. 초기 설정 | [예: 채널 연결 / API 키 입력 / 없음] |                 |             |      |
| 4. 핵심 액션 | [예: 첫 리포트 생성 / 첫 분석 실행]  |                 |             |      |
| 5. 가치 체감 | [예: 결과 확인 / 인사이트 발견]      |                 |             |      |
| 6. 재방문    | [예: 알림 → 재접속 / 주기적 사용]    |                 |             |      |

### 5.2 화면/인터페이스 목록

> 5.1 여정에서 도출된 화면을 나열한다. 라우트, 뷰, 팝업, 탭, CLI 커맨드 등 형태는 서비스에 맞게 자유롭게.

| 화면/인터페이스  | 역할              | 진입 경로 | 네비게이션 라벨             |
| ---------------- | ----------------- | --------- | --------------------------- |
| [예: 랜딩 → `/`] | 서비스 소개 + CTA | 직접 접근 | —                           |
| [자유롭게 추가]  |                   |           | → `brand_brief.md` 6.3 참조 |
|                  |                   |           |                             |

### 5.3 설계 검증 체크리스트

> 구조 설계 후 아래를 점검한다. 모두 채울 필요는 없고 서비스에 해당하는 것만.

- [ ] 사용자가 가입 후 핵심 가치를 체감하기까지 몇 단계인가? → 3단계 이내가 이상적
- [ ] 비로그인 상태에서도 가치를 미리 보여줄 수 있는가?
- [ ] 빈 상태(empty state)가 존재하는 화면은 어디인가? → 각각의 빈 상태 메시지 정의 필요
- [ ] 반복 사용하는 화면과 한 번만 거치는 화면이 구분되어 있는가?
- [ ] 모바일에서 핵심 플로우가 동작하는가? (모바일 필수인 서비스인 경우)

## 7. 랜딩 페이지 구현 스펙

> 카피와 구조는 `brand_brief.md` 8장을 그대로 따른다. 여기서는 기술적 구현만 정의한다.
> 랜딩이 필요 없는 아키타입(예: CLI 도구의 README, 스토어 리스팅)은 이 섹션을 해당 형태에 맞게 변환한다.
>
> **랜딩은 2단계로 배포한다.**
>
> - **Phase 1 (프리런치)**: 개발 초기에 배포. 웨이트리스트 수집 + 프리런치 콘텐츠 링크 용도.
> - **Phase 2 (풀런치)**: 제품 완성 후 업데이트. 실제 가입/결제 CTA + 기능 상세 추가.

### 7.1 Phase 1: 프리런치 랜딩 (D1~D2에 배포)

> 프리런치 콘텐츠(Threads, X, 커뮤니티 등)에 링크를 걸 수 있도록 빠르게 배포한다.
> 이 단계에서는 완성도보다 속도를 우선한다.

**필수 포함 요소**

| 순서 | 섹션                         | 카피 소스                         | 구현 노트                                             |
| ---- | ---------------------------- | --------------------------------- | ----------------------------------------------------- |
| 1    | 히어로 (헤드라인 + 서브카피) | → 브리프 8.1 프리런치 카피        | 핵심 메시지 전달                                      |
| 2    | Pain → Solution _(선택)_     | → 브리프 8.2 풀런치 카피에서 발췌 | 간결하게. 없어도 됨                                   |
| 3    | 웨이트리스트 CTA             | → 브리프 8.1 웨이트리스트 CTA     | 수집 도구: [Resend / Tally / Google Form / 자체 구현] |

**프리런치 CTA 설계**

| 항목           | 내용                                                                  |
| -------------- | --------------------------------------------------------------------- |
| CTA 문구       | [예: "런칭 알림 받기" / "무료 얼리엑세스 신청" / "웨이트리스트 등록"] |
| 수집 정보      | [예: 이메일만 / 이메일 + 이름 / 이메일 + 역할]                        |
| 수집 도구      | [예: Tally 임베드 / Resend audience / 자체 DB]                        |
| 등록 후 화면   | [예: "등록 완료! 런칭하면 가장 먼저 알려드릴게요." — 톤앤매너 반영]   |
| 등록자 수 표시 | [예: "현재 N명이 기다리고 있어요" 표시 여부]                          |

**SEO & 메타 (프리런치부터 적용)**

| 항목                   | 값                      | 소스                              |
| ---------------------- | ----------------------- | --------------------------------- |
| `<title>`              | [서비스명] — [태그라인] | → 브리프 4.1, 4.2                 |
| `meta description`     |                         | → 브리프 1.2 + 3.2 Primary 키워드 |
| OG title / description |                         | → 브리프 9.4                      |

### 7.2 Phase 2: 풀런치 랜딩 (제품 완성 후 업데이트)

> 제품이 동작하면 프리런치 랜딩을 풀런치 버전으로 업데이트한다.
> 웨이트리스트 CTA → 실제 가입/시작 CTA로 전환. 기능 상세, 가격, 스크린샷 추가.

**프리런치 대비 추가/변경되는 섹션**

| 순서 | 섹션             | 카피 소스                    | 변경 사항                                                      |
| ---- | ---------------- | ---------------------------- | -------------------------------------------------------------- |

| 3    | 기능 소개        | → 브리프 8.2 기능 소개       | **추가** — 실제 스크린샷/데모 포함                             |
| 4    | 가격 _(해당 시)_ | → 브리프 6.2 플랜            | **추가** — 플랜 비교 + 결제 연결                               |
| 5    | 신뢰 요소        | → 브리프 8.2 신뢰 요소       | **추가** — 베타 사용자 수, 후기 등                             |
| 6    | 최종 CTA         | → 브리프 8.2 최종 CTA        | CTA 전환 (히어로와 동일)                                       |

**웨이트리스트 → 기존 등록자 처리**

| 항목                      | 방법                                           |
| ------------------------- | ---------------------------------------------- |

## 9. 2주 스프린트 플랜

### 스프린트 설계 원칙

1. **Day 1은 셋업 + 프리런치 랜딩 배포**: 프로젝트 초기화와 동시에 프리런치 랜딩(웨이트리스트)을 배포한다. 프리런치 콘텐츠에 링크를 걸 수 있어야 하므로 OG/메타도 이때 세팅한다.
2. **핵심 가치 기능을 Week 1 안에 동작시킨다**: 부가 기능보다 사용자가 "아, 이게 이 서비스구나" 느끼는 기능을 우선
3. **Week 2 초반에 랜딩을 풀런치 버전으로 업데이트한다**: 기능 완성도를 보고 스크린샷, 기능 소개, 가격 섹션을 추가하고 CTA를 실제 가입으로 전환한다.
4. **마지막 2일은 폴리싱 + 런치 준비**: 빈 상태, 에러 처리, 반응형, 웨이트리스트 등록자 알림 준비
